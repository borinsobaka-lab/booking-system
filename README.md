# Запись на массаж — онлайн-запись + админка

Система онлайн-записи на услуги (массаж): витрина для клиентов и админка для
сотрудников. Фронтенд на React + Vite + TypeScript.

## Два режима хранения данных

- **Локальный** (по умолчанию, для разработки/демо): данные в браузере
  (`localStorage`). Ничего не публикуется, сервер не нужен.
- **Приватный бэкенд** (боевой): данные в **приватном** репозитории
  `booking-system-data` через **Cloudflare Worker** (`worker/`). Токен доступа
  живёт только на сервере; в публичном коде и браузерах его нет. Витрина получает
  только публичные данные (услуги, специалисты, занятость по времени — **без**
  имён, телефонов и почт), полные данные с контактами — только авторизованным
  сотрудникам. Настройка: **[worker/SETUP.md](worker/SETUP.md)**.

Режим переключается переменной сборки `VITE_API_BASE` (адрес Worker). Пусто ⇒
локальный режим. Слой данных изолирован (`src/db.ts`, `src/remote.ts`).

## Что уже есть

### Админка (`#/admin-panel` — отдельный адрес)
Вход для сотрудников доступен только по этому адресу и **нигде не связан
ссылками** из клиентской витрины (никаких кнопок «войти как сотрудник»).

- **Только вход, без регистрации.** Саморегистрации/создания администратора из
  интерфейса нет. **Суперадминистратор заводится вручную**: в боевом (remote)
  режиме — скриптом `worker/seed-owner.mjs` в приватный репозиторий; в локальном
  режиме для разработки/демо уже есть демо-владелец `demo` / `demo`.
- **Пользователи** (только для суперадминистратора): заведение **админов** и
  **мастеров**, выдача им логинов и паролей (пароль показывается один раз, можно
  сгенерировать и скопировать), смена паролей, удаление. Админы и мастера входят
  и видят записи, но **не** управляют пользователями.
- **Услуги**: название, описание, длительность, стоимость, картинка.
- **Специалисты**: аватар, имя, фамилия, роль и список выполняемых услуг.
- **Расписание**: у каждого специалиста на недельном таймлайне мышью
  «протягиваются» рабочие часы; отдельным режимом добавляются перерывы. Видно,
  какие дни рабочие, а какие выходные; выходной легко сделать рабочим.
- **Записи**: большой календарь на день по каждому специалисту — рабочие окна,
  перерывы и записи клиентов. Здесь же можно создать запись вручную или отменить
  существующую.

### Витрина клиента (`#/`)
Баннер бренда (аватар, название, адрес) и три пути записи:
1. **Мастер** → услуга → дата и время.
2. **Выбрать дату** → время (только когда есть свободный специалист) → услуга →
   мастер (недоступные в это время — заблюрены).
3. **Выбрать услугу** → мастер → дата и время.

Доступны только реально свободные даты/времена; после выбора запись
**подтверждается автоматически** (симуляция). Email-уведомления и полноценная
регистрация брони — следующий этап.

## Разработка

```bash
npm install
npm run dev        # http://localhost:5173/booking-system/  (локальный режим)
npm run build      # проверка типов + сборка в dist/
npm run test:e2e   # e2e-тесты Playwright

# Проверка бэкенда (Cloudflare Worker):
cd worker && node --test            # тесты логики API
cd worker && node mock-server.mjs  # локальный мок API (:8787)
```

Запуск фронтенда против локального мока API:
`VITE_API_BASE=http://localhost:8787 npm run dev`.

Приложение хэш-роутится: `#/` — витрина клиента, `#/admin-panel` — админка
(отдельный адрес, из витрины на неё нет ссылок).
